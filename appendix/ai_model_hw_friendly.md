## 1. なぜモデル側の設計がHLS実装効率を左右するのか（抽象原則）

### 1.1 抽象原則: 「同じ精度のモデル」でも実装コストは数倍〜数十倍変わる

同じタスク・同じ推論精度を達成できるモデルであっても、アーキテクチャ選択（演算子の種類、チャネル数の割り切れ方、分岐構造の有無）によって、HLS実装後のリソース使用量・レイテンシ・検証工数は大きく変わる。これは「モデル設計」と「ハードウェア実装」を分業した場合に起こる典型的な失敗——**ソフトウェア（学習）側では等価な精度でも、ハードウェア側では実装コストが等価ではない**——を避けるために最初に理解しておくべき前提である。**hardware-friendly（ハードウェア親和的）な設計**とは、この「精度を保ったまま実装コストを下げる」ための、モデル構造上の選択の集合を指す。

以下の5つの評価軸で、モデルのハードウェア親和性を判断する。

### 1.2 評価軸1: 演算強度（Compute / Memory 比、Roofline的視点）

**抽象原則**: HLSアクセラレータの性能は、多くの場合「演算器が動いている時間」ではなく「データがメモリ（オンチップBRAM/URAM、あるいはDDR）とやり取りされる帯域」で頭打ちになる。これはCPU/GPUのRooflineモデルと同じ理屈がFPGAにもそのまま当てはまる、という意味。1バイトのデータを読み込むごとに何回の積和演算ができるか（＝演算強度、arithmetic intensity）が低い層（例: 1x1 convで入出力チャンネルが少ない、あるいはFC層で重み再利用が効かない層）は、演算器をどれだけ並列化してもメモリ帯域で律速し、リソースを無駄に消費する。

**具体例**: SkyNetの3x3 depthwise convはチャンネルごとに独立した9回の積和（`dwconv3x3.cc`）で、1重みあたりの再利用回数が空間方向の出力画素数だけ確保できるため演算強度が比較的高い。一方1x1 pointwise convは「重み1個につき1回の積和」という最小の再利用単位になりやすく、SIMD/PE折り畳み（§8）でチャンネル方向に並列化して初めて演算器を稼働させ続けられる。**モデル設計段階で「1x1 convの前後にチャンネル数を極端に絞る層を挟まない」「極端に小さい層を単体で置かない（隣接層とfuseできる形にする）」といった配慮をしておくと、後段のHLS側でメモリ律速を避けやすくなる。**

### 1.3 評価軸2: データフローの規則性

**抽象原則**: HLSの`#pragma HLS DATAFLOW`は、複数の処理ステージを`hls::stream`で連結し、Vitis HLSに独立したハードウェアブロックとして並行実行させる仕組みである（FINNの標準構成、`claudes/finn_explained.md` §2）。この最適化が効くのは、**各ステージの入出力アクセスパターンが単調で予測可能（同じ順序で1回ずつ読み書きする）な場合**に限られる。逆に、ある層の出力を別の層が非連続な順序・複数回読み直す必要がある構造（例えばモデル側の都合で特徴マップを転置・reshapeしてから使うような設計）は、`hls::stream`の単方向・FIFO的なアクセスモデルと相性が悪く、追加のバッファリングやランダムアクセス可能なメモリへの経由が必要になり、パイプライン全体のII（Initiation Interval）を悪化させる。

**具体例**: SkyNetのdw3→dw6のskip接続では、PyTorch側の`ReorgLayer`（space-to-depth、`models.py:12-31`）が生成するチャンネル順序と、ハードウェアが要求する順序が食い違っていたため、活性化データではなく**重み側をオフラインで並べ替える**ことで帳尻を合わせた（`reorder_weight.cc`、"mysterious dw3" reorder、`claudes/skynet_explained.md` §6）。これは「モデル定義側の演算（reorg/transpose）をハードウェア実装側でそのまま素直に回路化する」のではなく、**モデル設計・重み前処理の段階でデータフローの規則性を回復する**汎用的なテクニックであり、今後モデルにreshape/transpose/permuteのような「配線を組み替えるだけの演算」が入る場合は、常に「活性化側で吸収するか、重み側で吸収するか」を検討すべきである。

### 1.4 評価軸3: 静的形状・静的制御フロー

**抽象原則**: HLSは基本的に「合成時に決まったループ境界・配列サイズ」を前提にした高効率化（`unroll`、`pipeline`、`array_partition`）を行う。実行時に分岐する形状（可変長シーケンス、データ依存の早期終了、動的な近傍探索など）は、HLSでは「常に最悪ケースの回路を敷いて、実行時にストールする」形でしか実装できず、面積効率が大きく落ちる。したがって**モデル設計段階で、実行時に形状が変わる要素（可変解像度、可変チャンネル数、データ依存分岐、動的トップK選択等）をできるだけ排除し、静的な形状・固定ループ境界だけで構成する**ことが、HLS実装コストを大きく下げる。

**具体例**: SkyNet・FINNいずれも「1つの固定ネットワークを1回だけ合成する」設計であり、レイヤーごとのループ境界・チャンネル数はすべてコンパイル時定数（`net_hls.cc`の`CI_N`/`CO_N`、`mvau.hpp`の`MatrixH`/`MatrixW`テンプレート引数）である。VTAだけは例外的に「命令列で実行時に形状を可変にする」設計だが、これは汎用アクセラレータであるための代償であり、単一ネットワーク専用設計では過剰投資になる（`claudes/tvm-vta_explained.md` §10）。**Transformer的な構造を将来検討する場合も、Attentionの系列長・ヘッド数を実行時可変にしない（固定長にパディング/切り詰めしてから合成する）ことが、この原則の直接的な適用例になる。**

### 1.5 評価軸4: 量子化耐性

**抽象原則**: モデルが低ビット幅・整数演算にどれだけ耐えられるか（精度劣化が許容範囲に収まるか）は、演算子の種類・チャンネル間の値のばらつき・活性化関数の出力レンジに強く依存する。量子化耐性の低い演算（Softmax、LayerNorm、大きな値域を持つ層）を含むモデルは、たとえ他の層が量子化フレンドリーでも、その一箇所のために広いビット幅・浮動小数点演算・複雑な近似回路を持ち込む必要が生じ、ハードウェア全体の複雑さを押し上げる。**モデル設計段階で「量子化に強い演算子だけで構成する」ことは、単に精度の問題ではなく、ハードウェアの均質性（＝実装・検証コスト）を左右する。**（詳細は§2, §3）

### 1.6 評価軸5: パラメータ・中間活性化のオンチップ収容可能性

**抽象原則**: DDRを使わない（オンチップ完結）設計を目指す場合、モデル全体の重み（量子化後）と、スキップ接続等で保持が必要な中間活性化の総量が、ターゲットデバイスのBRAM/URAM総量に収まるかどうかが、そもそも設計が成立するかの絶対条件になる。FINNも標準はオンチップ完結だが、モデルが大きすぎる場合は「重みをストリームから読む」バリアント（`Matrix_Vector_Activate_Stream_Batch`等）に逃げる設計を用意しており（`claudes/finn_explained.md` §2）、この制約は現実に無視できない。**モデル設計段階で、チャンネル数・層数を増やす前に「量子化後の重み総量×活性化バッファ総量」を電卓で見積もっておく**ことが、後戻りのないアクセラレータ開発の出発点になる。

### 1.7 5軸の一覧表

| 評価軸 | 悪い例（hardware-unfriendly） | 良い例（hardware-friendly） |
|---|---|---|
| 演算強度 | 極小チャンネルの1x1 convを単体で挟む、重み再利用の乏しい層 | チャンネル数を十分に確保し、隣接層とfuse可能な構成にする |
| データフローの規則性 | 実行時のreshape/transpose/gather、非単調なメモリアクセス | 層順に一方向ストリームで流せる構成、並べ替えは重み側で吸収 |
| 静的形状・制御フロー | 可変長系列、データ依存分岐、動的トップK | 固定解像度・固定チャンネル数・固定ループ境界 |
| 量子化耐性 | Softmax/LayerNorm/GELU、外れ値の大きい層 | ReLU系、BN-fusable線形演算、対称レンジの層 |
| オンチップ収容性 | 見積りなしにチャンネル数・層数を増やす | 量子化後サイズをBRAM/URAM予算と最初に照合する |

## 2. 量子化（Quantization）

### 2.1 per-tensor vs per-channel

**抽象原則**: 量子化スケール（あるいは`ap_fixed`の小数点位置）を「テンソル全体で1つ」にするか「チャンネルごとに1つ」にするかは、精度とハードウェアコストのトレードオフである。per-tensor量子化はスケールが1つなので、積和のアキュムレータに対して1回だけリスケールすればよく回路が単純になる。per-channel量子化は、重みのチャンネルごとのレンジのばらつきを吸収でき精度は上がるが、**積和後にチャンネルごとに異なるスケールを掛ける必要があり、requantizeのタイミング・回路規模が増える**。

Googleの量子化ホワイトペーパー（arXiv:1806.08342）は「重みはper-channel、活性化はper-layer（per-tensor）」を推奨構成としており、これは実務上の定石になっている——**活性化側までper-channelにすると、conv/matmulの内積計算そのものが複雑化する**ためである（活性化は入力チャンネル方向に累積されるため、チャンネルごとに異なるスケールを持たせると積和の各項を都度リスケールしてから足す必要が生じる）。

**具体例（`ap_fixed`との対応）**: SkyNetは実質per-tensor相当の設計で、`FIX_FM`/`FIX_WT`という単一の型を全チャンネル・全層で共通に使う（`net_hls.h:45-67`）。per-channelを実現したい場合、HLS側では「チャンネルごとに異なる`ap_fixed`型」を持つのではなく、**共通の広いビット幅アキュムレータに一旦積和した後、チャンネルインデックスで引く配列（重み配列と同じ構造の`scale[channel]`配列、あるいはFINNの`ThresholdsActivation`のようなper-channel閾値配列）でrequantizeする**、という設計にするのが自然である（`activations.hpp`の`m_thresholds[PE][NF][NumTH]`は既にPE軸＝チャンネル軸を持つ設計になっている）。

### 2.2 ビット幅選択と`ap_fixed`/`ap_int`との対応

**抽象原則**: `ap_fixed<W,I,Q,O>`（W=総ビット幅、I=整数部ビット幅、Q=丸めモード、O=飽和モード）は、量子化の「スケール」を暗黙の2進小数点位置として型に埋め込む、Xilinx HLS特有の量子化表現である。これは一般的な「整数+zero-point+scale」というaffine量子化とは異なり、**ゼロ点を持たない対称固定小数点**であるため、ReLU系の非負出力・対称レンジの重み分布と相性が良い。モデル設計段階でビット幅を決める際は、単に「8bit/16bit」という総ビット幅だけでなく、**整数部と小数部の配分（＝表現可能な値域とLSB分解能のトレードオフ）**を、層ごとの値域の実測（学習後のヒストグラム確認）に基づいて決めるべきである。

**具体例**: SkyNetは`FIX_FM = ap_fixed<9,3,AP_RND,AP_SAT>`（活性化: 値域約[-4,4)、LSB=1/64）、`FIX_WT = ap_fixed<11,4,AP_RND,AP_SAT>`（重み: 値域約[-8,8)、LSB=1/128）、`FIX_FM_acc = ap_fixed<12,4,...>`（アキュムレータ、活性化より小数部を1bit広げて丸め誤差を抑制）という型階層を採用している（`net_hls.h:45-67`）。これはMLSys'20論文Table 7の総当たり実験（float32ベースライン IoU 0.741 → 9bit FM/11bit weightでIoU 0.727、-1.4%）で選定された値であり、**「まず学習側でビット幅を総当たり探索し、その結果をHLSの型定義（コンパイル時定数）としてそのまま埋め込む」**という、モデル設計とHLS実装を1対1に対応づける手法が実例として確認できる（`claudes/skynet_explained.md` §3, §10）。VTA/FINNはより一般化して`ap_int<N>`のNをテンプレートパラメータとして注入する設計（`TSrcI`/`TDstI`/`TWeightI`, `claudes/finn_explained.md` §4）を取っており、複数ビット幅構成を1つのHLSコードベースで使い回したい場合はこちらが参考になる。

### 2.3 QAT (Quantization-Aware Training) vs PTQ (Post-Training Quantization)

**抽象原則**: PTQ（学習後に量子化パラメータだけを決める）は工数が小さいが、低ビット幅（8bit以下）や特殊な演算子を含むモデルでは精度劣化が大きくなりやすい。QAT（学習中に疑似量子化＝fake quantizationを挟んで、量子化誤差を勾配に反映させながら学習する）は工数が大きいが、低ビット幅でも精度を保ちやすい。**FPGA向けの低ビット幅（4〜9bit程度）を狙うモデルでは、QATがほぼ前提になる**——SkyNetもFINNも、学習側で量子化を意識したビット幅探索・学習（fake quantization）を行っており、HLS側は「学習済みの量子化パラメータをそのまま型・配列に落とし込むだけ」という役割分担になっている（`claudes/skynet_explained.md` §3、`claudes/finn_explained.md` §9 FINN-Rの"Streamlining"パス）。

**具体例**: 実務上のワークフローとしては、(1) PyTorch/Brevitas（FINNの標準フロントエンド）やQKeras（hls4ml向け）のような量子化認識学習ライブラリでビット幅を意識した学習を行う、(2) 学習後にBatchNormを畳み込みへfuse（§5）、(3) 量子化パラメータ（スケール/ゼロ点、あるいは閾値配列）をオフラインツールで算出、(4) それをHLSソースの`const`配列・型定義として書き出す、という4段階が定石になる。この一連のワークフローを支えるオフライン変換スクリプト自体（SkyNetの`reorder_weight.cc`に相当するもの）は、**モデル設計の一部として最初から用意しておくべきツールチェーンである**、という認識が重要——HLS実装を始めてから量子化パラメータの受け渡し方法を後付けで考えると、型設計のやり直しが発生しやすい。

### 2.4 再量子化（requantization）をモデル設計段階でどう考慮するか

**抽象原則**: 層と層の間で必ず発生する「あるビット幅の中間表現を、次の層が期待するビット幅へ変換する」処理（requantize）は、モデル設計段階で以下を決めておくとHLS実装が大きく単純化される。
1. **各層の出力レンジ（活性化関数の値域）を、次層が扱いやすい範囲に事前に制約しておく**（例: ReLU6のように上限を明示的にクリップする活性化を選ぶ、§3.1）。
2. **BatchNorm等のスケーリング層を、量子化層の直前に配置し、後でrequantize処理へ畳み込めるようにしておく**（FINN-R論文の"Streamlining"、`claudes/finn_explained.md` §9）。
3. **層タイプ（畳み込みの累積深さ）ごとに丸めのタイミングを揃える**——SkyNetは深さwise convとpointwise convで丸めのタイミングが異なる（`dwconv3x3.cc`はタップごとに9回丸め、`conv1x1.cc`は16タップの加算木の最後に1回だけ丸め）という非対称性を実装上たまたま持ってしまっており、これは意図的な設計ではなく「反面教師」として明記されている（`claudes/skynet_explained.md` §4, §11）。**新規モデル設計では、この丸めタイミングの非対称性を最初から意識的に統一する**べきである。

### 2.5 閾値ベース（乗算器を使わない）実装に適したモデル設計

**抽象原則**: FINNの`ThresholdsActivation`（`activations.hpp:200-221`）は、BatchNorm・非線形活性化・requantizeスケールを学習後にオフラインで1本の閾値配列へ畳み込んでおき、ハードウェア側は「アキュムレータ値がいくつの閾値を上回ったか」を数えるだけ（比較器＋1bitカウンタの加算木）で済ませる、乗算器・シフタ不要のrequantize方式である（`claudes/finn_explained.md` §5）。この方式が効果的なのは**出力ビット幅が小さい場合に限られる**——`NumTH = 2^W - 1`個の閾値・比較器が必要になり、出力ビット幅に対して指数的に増加するため、8bit程度まではLUTコストが現実的だが、12〜16bitの中間表現にそのまま適用するのは非現実的（`claudes/finn_explained.md` §5, §11）。

**モデル設計側の含意**: 「閾値方式を使いたい層」を見越して、**中間活性化のビット幅を意識的に低く抑える（4〜8bit程度）設計にしておく**と、乗算器レスのrequantizeが選択肢に入る。逆に、再構成誤差を扱うAutoEncoderの出力層のように、広いダイナミックレンジ・高ビット幅が必要な層は、最初から閾値方式の対象外と割り切り、算術的なrequantize（シフト+クリップ、あるいは`ap_fixed`の暗黙リスケール）に回す、という**層ごとの使い分けをモデル設計時点で決めておく**ことが、後段のHLS実装を大きく単純化する。

| 方式 | 実装コスト | 精度制御性 | 適用が現実的なビット幅 | 代表実装 |
|---|---|---|---|---|
| `ap_fixed`暗黙リスケール | 最小（言語機能に依存） | 型定義固定、後から変更しにくい | 任意 | SkyNet |
| 明示的シフト+クリップ（整数演算） | 小〜中 | 実行時パラメータ化しやすい | 任意（特に高ビット幅向き） | VTA |
| 閾値比較（乗算器レス） | 低ビット幅では最小、高ビット幅では爆発 | 任意の単調関数を表現可能 | 〜8bit程度 | FINN |

## 3. 演算子選択（Operator Selection）

### 3.1 ReLU/BN-fusable活性化を優先する理由

**抽象原則**: 活性化関数は「その出力を次の量子化ステップへどう橋渡しするか」まで含めてハードウェアコストが決まる。ReLU（および出力上限を明示するReLU6等）は、(a) 単純な比較器1個で実装できる、(b) 出力が非負（あるいは有限区間）に制約されるため量子化ビット幅を節約できる、(c) 直前のBatchNormと線形演算として畳み込める（§5）、という3つの理由でハードウェアフレンドリーとされる。**SkyNetがReLU（[0,∞)）ではなくReLU6（[0,6]）を採用した理由は精度目的だけでなく、「出力レンジが[0,6]に制限されるため元のReLUよりも少ないビット数で表現できる」という、量子化を見据えたハードウェア都合の活性化関数選択として論文に明記されている**（MLSys'20論文、`claudes/skynet_explained.md` §10）。これは「精度が同じなら、値域を絞れる活性化関数を選ぶ」という、モデル設計段階でのhardware-friendly判断の典型例である。

### 3.2 Softmax/LayerNorm/GELU等の非線形処理がFPGA上で厄介な理由と代替

**抽象原則**: Softmax（指数関数＋総和による正規化）、LayerNorm（バッチではなく特徴次元方向の平均・分散を都度計算する必要がある正規化）、GELU（誤差関数を含む滑らかな非線形）は、いずれも以下の共通した困難を持つ。
1. **超越関数（exp, erf等）を含み、固定小数点での高精度近似には多項式近似・LUT・CORDIC等の追加ロジックが必要**になる。
2. **総和・平均・分散といった、その場で完結しない「全要素にわたる集約」を要求する**ため、ストリーミング処理（1要素ずつ流し切る設計）と相性が悪く、全要素をバッファしてから2パス目で処理する、という構造を強いられやすい。
3. 出力レンジが動的（入力に依存して変わる）なため、固定的な`ap_fixed`スケールでの量子化が難しい。

**代替・回避の設計判断**:
- SkyNetは検出ヘッドのsigmoid/expによるボックスデコードを**FPGA上では行わず、ホスト（PS/ARM）側にfloatで逃がす**という判断をしている（`net_hls.cc`、`claudes/skynet_explained.md` §8, §11）。「非線形な後処理はPL側で固定小数点実装せず、PS/ホスト側に逃がす」という判断は、AutoEncoderの損失計算や、稀にしか通らない非線形演算がある場合のリソース対効果を考えるうえでそのまま参考になる。
- 正規化（画素値の正規化等）はLUT化（SkyNetの`img_norm_ch[256]`, `net_hls.cc:491-508`）できる場合はLUT化し、実行時の割り算・浮動小数点演算を排除する。
- LayerNorm/Softmaxを含むTransformer的な構造をどうしても使う場合は、(a) LayerNormをBatchNorm的な「学習後に統計量が固定される正規化」（例えば推論時は移動平均統計を使うことで、実行時の平均・分散計算を回避する）に近い形へ寄せる、(b) Softmaxの総和対象範囲を小さく（ヘッド数・系列長を絞る）保つ、といった**モデル構造側の制約でハードウェア負荷を下げる**アプローチが現実的である。この領域は本リポジトリの先行調査（SkyNet/FINN/VTA）に直接の実装例がなく、**hls4ml等の別プロジェクトでのTransformer実装例を今後別途調査する必要がある**、という点は正直に明記しておく。

### 3.3 Depthwise Separable Convolutionのメリット・デメリット

**抽象原則**: 通常の畳み込み（全チャンネル間で積和する）を、depthwise conv（チャンネルごとに独立した空間フィルタ）+ pointwise conv（1x1、チャンネル間だけを混合）に分解するdepthwise separable convは、パラメータ数・演算量を`1/Cout + 1/K²`程度に削減できる（MobileNet系で広く使われる設計）。ハードウェア観点では以下のメリット・デメリットがある。

**メリット**:
- 演算量・パラメータ量の削減はそのままBRAM/URAM消費量・DSP使用量の削減に直結する。
- FINNの`vvau.hpp`（VVAU, Vector-Vector Activate Unit）のように、**チャンネル間reductionがないため`SIMD=1`固定で実装が単純化される**（`claudes/finn_explained.md` §8）——通常convのMVAU（`mvau.hpp`）が持つ「入力チャンネル軸の内積」というreduction構造そのものが不要になる。

**デメリット**:
- pointwise（1x1）conv側は§1.2で述べた「演算強度が低い層」になりやすく、チャンネル数が小さいとメモリ律速・PE稼働率低下を招きやすい。
- depthwise conv自体はチャンネル方向の並列度（PE数）をそのまま出力チャンネル数に対応させる必要があり、**チャンネル数がPE数の倍数でないと端数処理（ゼロ埋め等）が必要になる**（§4.1）。

**具体例**: SkyNetの`conv_dw`ブロック（`models.py:49-58`: DW-Conv3x3 + BN + ReLU6 + PW-Conv1x1 + BN + ReLU6）は、この分解を「Bundle」という最小構成単位として定義し（DAC'19論文、`claudes/skynet_explained.md` §10）、HLS側でも`DW_CONV_3x3`と`CONV_1x1`という2種類のPE関数として明確に分離実装している（`dwconv3x3.cc`, `conv1x1.cc`）。**このように、モデル設計段階で「depthwise」と「pointwise」を明示的に別ブロックとして扱っておくと、HLS側でも2種類のPE関数へ素直に対応づけられる**、という設計上の対応関係が確認できる。

### 3.4 対比表: preferされる演算 vs ハードウェア非friendlyな演算

| カテゴリ | Hardware-friendly（優先） | Hardware-unfriendly（要注意・代替検討） | 理由 |
|---|---|---|---|
| 活性化関数 | ReLU, ReLU6, LeakyReLU（固定シフト版） | Sigmoid, Tanh, GELU, Swish | 超越関数近似が必要、出力レンジが不定 |
| 正規化 | BatchNorm（推論時は線形演算にfuse可能） | LayerNorm, InstanceNorm | 実行時に統計量計算が必要、ストリーミング非親和 |
| 分類ヘッド出力 | 単純な閾値判定・argmax | Softmax（確率化） | 指数関数＋総和、動的レンジ |
| 畳み込み | 通常conv, depthwise separable conv, 小カーネル(1x1/3x3) | 大カーネル(7x7以上), dilated conv（大dilation） | ラインバッファ・ウィンドウサイズが線形〜二次に増大 |
| アップサンプリング | 最近傍補間+conv, PixelShuffle(space-to-depth系) | Transposed Convolution（ストライド>1） | ゼロ挿入によるスパース演算、チェッカーボードアーティファクト（§7） |
| 演算子の入出力形状 | 固定形状、チャンネル数がPE/SIMD数の倍数 | 動的形状、割り切れないチャンネル数 | 静的合成の前提が崩れる、端数処理が必要 |
| 集約演算 | チャンネル方向の固定木（加算木） | 系列全体・画像全体にわたる集約（Global Attention等） | バッファリング・多パス処理が必要 |

## 4. 次元・形状設計（Dimension & Shape Design）

### 4.1 チャネル数・カーネルサイズをPE数/SIMD数で割り切れるようにする理由

**抽象原則**: HLSでのチャンネル方向並列化は、`#pragma HLS array_partition`+`unroll`（SkyNet流）あるいはSIMD/PEテンプレートパラメータ（FINN流）によって、**「1サイクルで同時に処理するチャンネル数」を固定のハードウェア並列度として実装する**。このとき、モデルの実際のチャンネル数がその並列度の倍数でない場合、(a) 端数チャンネルをゼロ埋めして並列度の倍数に切り上げる、(b) 最後のグループだけ可変長ループにする、のいずれかの対処が必要になり、後者は静的形状の原則（§1.4）を破ってII悪化・制御ロジック増大を招く。**したがってモデル設計段階で、チャンネル数を採用予定のPE数/SIMD数の倍数に揃えておく**ことが望ましい。

**具体例**: SkyNetは32チャンネル単位のPE配列（32出力ch並列×16入力ch積和木、`conv1x1.cc:91-142`）を採用しており、入力3ch（dw1の入力）や48ch（dw1/dw2出力）のような32の倍数でないチャンネル数は、**オフラインツール（`reorder_weight.cc:172-204`）でゼロパディングして32の倍数へ揃えている**（3→32ch, 48→64ch）。これは「モデル定義側のチャンネル数をそのまま使う」のではなく、**ハードウェアの並列度に合わせてモデル側（重み配列）を後処理で拡張する**という現実解であり、新規モデル設計では**最初からPE数の倍数になるチャンネル数を選ぶ**方が、パディング・端数処理の分だけ効率が良い（無駄な演算・面積を減らせる）。FINNのSIMD/PE（`mvau.hpp`の`NF = MatrixH/PE`, `SF = MatrixW/SIMD`）も同様に、`MatrixH`/`MatrixW`が`PE`/`SIMD`で割り切れることを前提にした設計になっている。

### 4.2 パディングと境界処理の複雑さ

**抽象原則**: 畳み込みの境界（画像端）処理は、内部と同じ回路をそのまま流用できず、`if`分岐やゼロ埋めロジックを追加で必要とする。カーネルサイズが大きい・ストライドが不規則・非対称パディングを使う設計は、この境界処理ロジックを増やし、パイプラインのII・LUT消費を悪化させる。**モデル設計段階で、カーネルサイズを小さく（3x3や1x1中心に）保ち、パディングは対称（`same`パディング相当）に統一し、ストライドは2の累乗（1, 2, 4）に揃える**ことで、境界処理を単純化できる。

**具体例**: FINNの`ConvolutionInputGenerator`（`slidingwindow.h:159-260`）はラインバッファ方式で境界を扱うが、カーネルサイズ・ストライドがテンプレートパラメータとして固定されることを前提に、`number_blocks = ConvKernelDim/Stride + 1`というリングバッファサイズを静的に決めている（`claudes/finn_explained.md` §7）。ストライドやカーネルサイズが層ごとに大きくばらつくモデルは、層ごとに全く異なるバッファ構造を用意する必要があり実装・検証コストが増える——**モデル設計段階でカーネルサイズ・ストライドのバリエーションを絞っておく（例えば3x3ストライド1と2x2プーリングだけで構成する、SkyNetの構成がまさにこれ）**ことが、HLS側の再利用性を高める。

### 4.3 入力解像度とストリーミング処理の関係

**抽象原則**: DDRを使わないオンチップ完結設計（`hls::stream`+`#pragma HLS DATAFLOW`によるストリーミング）では、特徴マップ全体ではなく「カーネル高さ+α」の行数だけをラインバッファとしてオンチップに保持し、入力を最初から最後まで一直線に流し切る（§4.2のFINN方式）。この方式が効率よく機能するには、**入力解像度が小さすぎない（ストリーム立ち上がりのレイテンシが支配的にならない）一方、大きすぎない（ラインバッファ1本のサイズ＝解像度の幅×チャンネル数がBRAM予算を超えない）**という、程よいレンジに収まっている必要がある。モデル設計段階で入力解像度・初期層のチャンネル数を決める際は、**「ラインバッファ1行分のサイズ×保持行数」が現実的なBRAM容量に収まるか**を電卓で確認しておくべきである（§1.6のオンチップ収容可能性の一部）。

SkyNetのようにDDRを使うタイル方式（44x84タイル+ping-pongダブルバッファ、`net_hls.cc:837-844`）は、オンチップに収まらない解像度・チャンネル数を扱うための代替手段だが、DDRなし設計を志向する場合はFINNのライン・バッファ方式（タイル境界という概念自体を持たない一直線のストリーム処理）が優先される（`claudes/design_philosophy.md` §5, §8）。

## 5. 演算子融合（Operator Fusion）

**抽象原則**: 演算子融合（Operator Fusion / Layer Fusion）とは、複数の連続する演算（典型的にはConv→BatchNorm→Activation）を、推論時には1つの線形変換＋非線形変換へまとめてしまう最適化である。これは単なる「計算量削減」ではなく、**HLS実装においては「ステージ数・`hls::stream`（FIFO）の本数・パイプラインの段数」を直接削減する**という、ハードウェア実装コストに直結する意味を持つ。DATAFLOW領域では1ステージ＝1つの独立したハードウェアブロック（＋前後をつなぐFIFO）になるため、融合によってステージ数を減らすことは、面積（FIFOバッファ分のBRAM/LUTRAM）とレイテンシ（ステージ間の受け渡しオーバーヘッド）の両方を削減する。

### 5.1 Conv + BN + Activationの融合

BatchNorm（推論時は`y = γ(x-μ)/√(σ²+ε) + β`という各チャンネルに対する固定のアフィン変換）は、直前のConv/Linear層の重み・バイアスへ数式的に畳み込める（`W' = W・γ/√(σ²+ε)`, `b' = (b-μ)・γ/√(σ²+ε) + β`）。これにより、推論時のハードウェアからBatchNorm層そのものを消し去り、Conv層の出力に対してそのまま次の活性化・requantizeへ直結できる。

**具体例**: SkyNet・FINNいずれも、BatchNormは学習後にオフラインでConv/Linearの重み・バイアスへfuse済みという前提でHLS実装が構成されている（SkyNetの`weights_floating.bin`は"layer fusion to merge both parameters from Conv and its successive BN offline"と論文に明記、`claudes/skynet_explained.md` §3）。FINN-R論文の"Streamlining"パスも同じ操作を「量子化層の直前にあるスケーリング層を1つの線形変換へ畳み込み、それを多段閾値へ吸収する」という形で一般化している（`claudes/finn_explained.md` §9）。**新規モデル設計でも、BatchNormを単体レイヤーとしてHLS化することは基本的に避け、常に前段の線形演算へfuseする前提でモデル・オフライン変換パイプラインを組む**ことが定石になる。

### 5.2 融合がHLS側の段数・FIFO数削減に直結する具体的な理由

DATAFLOW領域で「Conv → BN → ReLU」を3つの独立ステージとして実装すると、Conv出力用・BN出力用の2本の`hls::stream`FIFOが余分に必要になり、パイプライン全体のレイテンシもステージ数分だけ増える。融合により「Conv（BN込みの重み・バイアスを使用）→ ReLU（+ requantize）」の2ステージ、あるいはさらにrequantizeまで1つの代入文（`ap_fixed`のキャストやFINNの`ThresholdsActivation`）に統合すれば実質1ステージへ圧縮できる。**「融合できる境界をなるべく多く作る」ことは、モデル構造の設計段階で、活性化関数・正規化層の配置を意識的に選ぶことで実現できる**——例えば正規化層を畳み込みの直後に固定して置く（分岐の途中やskip接続の合流点の後に挟まない）、活性化関数をBN-fusable演算に限定する（§3.1）、といった配慮がこれにあたる。

## 6. スキップ接続・分岐構造

### 6.1 抽象原則: 分岐はバッファリング・レイテンシ整合のコストを生む

スキップ（バイパス）接続は、モデルの表現力・精度に大きく寄与する一方、ハードウェア実装では「メインパス（畳み込み等を経由する側）」と「バイパスパス（素通りする側）」で処理レイテンシが異なるため、**合流点で両者のタイミングを合わせるためのバッファ（FIFO）が必要になる**、という構造的なコストを常に伴う。この問題は、SkyNet・VTA・FINNの3モデルいずれにおいても——実装アプローチは異なるものの——共通して現れており、かつ**FINNの`DuplicateStreams`/`AddStreams`（ResNet-50のskip接続を想定した専用プリミティブ）のコード自体にも、バイパスFIFOの深さを決定する明示的な規約は見当たらなかった**、という点が先行調査で確認された未解決課題である（`claudes/finn_explained.md` §6, `claudes/design_philosophy.md` §3, §9）。

### 6.2 各モデルの分岐・合流の実装方式（先行調査からの引用）

| モデル | 分岐（duplicate） | 合流（merge） | バッファリング方式 |
|---|---|---|---|
| SkyNet | skip元をDDR上の固定オフセットへ書き出し | 後続レイヤーがDDRから読み直す＋オンザフライでビット再配置 | DDRを大容量スクラッチパッドとして使用、レイテンシ整合は暗黙的（DDR経由なのでタイミングクリティカルではない） |
| VTA | 専用ハードウェアなし | 汎用`ALU_ADD`命令で`acc_mem`上の2インデックスを加算 | オンチップ`acc_mem`をスクラッチとして使用、タイミング管理はコンパイラ（命令スケジューリング）の責務 |
| FINN | `DuplicateStreams`（1入力→2出力ストリームへ同時write） | `AddStreams`（チャンネルごとの単純加算） | オンチップストリームのまま、**FIFO深さの明示的な指定はコード上見当たらない（未解決点として明記済み）** |

### 6.3 モデル側で緩和する設計（バイパスFIFO問題への対処）

HLS実装側の工夫（FIFOの深さを実測・見積もって明示的に`#pragma HLS STREAM depth=`で指定する等）だけに頼らず、**モデルアーキテクチャ設計の段階で分岐構造そのものを単純化しておく**ことで、この問題を根本的に緩和できる。以下は抽象原則→具体的なモデル設計指針の対応である。

- **抽象原則: 分岐点の数を減らす。** スキップ接続を持つブロックの数が増えるほど、管理すべきバイパスFIFOの本数・見積もりの手間が線形に増える。**具体策**: U-Net的に「各解像度段で1本ずつ」のようにスキップを疎に配置し、DenseNet的な「全層間でスキップ」のような密な接続は、精度上の利点とハードウェア複雑度のトレードオフを見て慎重に採否を判断する。
- **抽象原則: バイパス経路（メインパスとの合流までの距離）を短くする。** バイパス元からバイパス先までの間にメインパス側の層が多いほど、メインパス側のレイテンシが伸び、バイパスFIFOに必要な深さ（＝BRAM消費量）が増える。**具体策**: スキップ元を合流点の直前の解像度段に置く（SkyNetのdw3→dw6のように、間に挟む層数を最小限にする）、あるいはEncoder-Decoder構造では対称な位置（同じ解像度段）同士をスキップさせ、非対称な長距離スキップを避ける。
- **抽象原則: バイパス経路上での追加演算を避ける。** バイパス側に何らかの変換（チャンネル数調整の1x1 conv等）を挟むと、そのバイパス側処理自体のレイテンシも見積もり対象に加わり、問題が「メインパス vs 素通り」の単純な2択ではなくなる。**具体策**: 可能であればバイパス元とバイパス先のチャンネル数・解像度を一致させ、バイパス経路には変換を挟まない設計にする（一致させられない場合は、SkyNetのように活性化データではなく重み側のオフライン並べ替えで吸収する、§1.3参照）。
- **抽象原則: 合流演算を単純な加算・連結に限定する。** FINNの`AddStreams`が単純なチャンネルごとの加算に限定されているように、合流点での演算を「element-wise加算」か「channel concat」のどちらかに限定しておくと、HLS側の合流ロジックが定型化でき、可搬性の高いテンプレート関数（MVAU/VVAUに対する`AddStreams`のような汎用部品）として1回実装すれば使い回せる。

### 6.4 実務上の指針（現時点でのベストプラクティス、正直な限界の明記）

上記のモデル側の緩和策を講じてもなお、**バイパスFIFOの必要深さそのものは「メインパス側のレイテンシ（Conv+活性化+requantizeの合計サイクル数）を見積もった上で、それを吸収できるだけの深さを明示的に設定する」という、実装側の作業が最終的には必要になる**——これは3モデルのいずれも自動解決してくれない設計判断であり、モデル設計段階での緩和は「見積もり・実装の手間を減らす」ものであって「不要にする」ものではない、という点を正直に記しておく。ユーザーが実際にAutoEncoderのスキップ接続を設計する際は、(1) メインパス側のレイヤーごとのレイテンシ見積もり表を作る、(2) その合計に安全マージンを乗せた深さでバイパスFIFOを確保する、(3) C-Simulation/RTLシミュレーションで実際にストール（バックプレッシャ）が発生しないか検証する、という手順を踏むことを推奨する。

## 7. AutoEncoder / Encoder-Decoder特有の考慮

### 7.1 Transposed Convolution / Upsamplingのハードウェア実装コスト

**抽象原則**: AutoEncoderのDecoder側で解像度を戻す演算として、Transposed Convolution（逆畳み込み、strideを持つ場合は出力側にゼロを挿入してから通常の畳み込みを行う）がよく使われるが、これは**「入力画素の間にゼロを挿入してから畳み込む」という構造上、演算のかなりの割合が「ゼロとの積」という無駄な計算になる**（stride=2なら理論上最大75%が無効な演算）。またゼロ挿入パターンとカーネルサイズの組み合わせによっては出力に周期的なムラ（チェッカーボードアーティファクト、これは学習側の一般的な既知課題であり、ODIN論文等で指摘されている）が生じることもある。ハードウェア側でこれを愚直に実装すると、(a) スパースな演算パターンをそのまま回路化すると演算器の稼働率が落ちる、(b) ゼロ挿入・出力位置計算のための追加の制御ロジックが必要、という二重のコストがかかる。

**代替設計**: 学習コミュニティで広く知られる回避策として、**「最近傍補間（あるいはPixelShuffle系のsub-pixel convolution）による単純なアップサンプリング + 通常の畳み込み（stride=1）」**という構成に置き換える方法がある。これは(1)アップサンプリングとconvを分離することで、アップサンプリング自体を「単純な画素複製・ライン重複読み出し」というメモリアクセスパターンに帰着でき、ハードウェア実装がストリーミングと親和的になる、(2)畳み込み自体は通常のstride=1畳み込みなので、Encoder側で使うのと同じPE/ラインバッファ構造をそのまま再利用できる（§7.2）、という2つの利点がある。**ハードウェア実装コストを重視する場合、モデル設計段階でTransposed Convolutionではなくこの構成を第一候補として検討すべき**である。ただし本リポジトリの先行調査（SkyNet/VTA/FINN）にはDecoder側（転置畳み込み/アップサンプリング）の直接的な実装例が薄く（FINNには`upsample.hpp`が存在するが、先行調査では概要レベルの言及にとどまっている、`claudes/design_philosophy.md` §9）、**転置畳み込みそのものをHLSで実装する場合の定型パターンは、本ドキュメントの範囲では実例に基づいた確証が取れていない**点を明記しておく。

### 7.2 対称構造の再利用可能性

**抽象原則**: AutoEncoderの多くはEncoder/Decoderが対称な層構成（チャンネル数・解像度が逆順で対応する）を取る。これはモデルの表現力上の理由だけでなく、**ハードウェア実装においては「Encoder側で合成したPE関数・ラインバッファ構造を、Decoder側でもチャンネル数・カーネルサイズが一致する限りそのまま再利用できる」という大きな利点**を持つ。§8で述べる「レイヤー横断的なPE共有」を、単一パス内の複数レイヤーだけでなく、**Encoder/Decoderという構造的に対称な2つのパスの間でも適用できる**、という発想がここでの核心である。

**具体例（モデル設計指針として）**: Encoder側の各段が「Conv(stride=2, ダウンサンプリング) → depthwise separable conv × N」という構成なら、Decoder側は「Upsample(最近傍) → depthwise separable conv × N（Encoder側と同じPE構成が使える形状に揃える）」という構成にし、**チャンネル数の遷移をEncoder側と同じ刻み幅（例: 32の倍数）で逆順にたどる**ように設計しておくと、SkyNet流の`ALLOCATION limit=1`（§8）や、FINNのMVAU/VVAUテンプレートのインスタンス化パラメータを、Encoder/Decoderの両方で使い回せる可能性が高まる。これはモデル精度上の制約ではなく、**ハードウェア再利用性を最大化するための、モデル構造上の意図的な対称性の維持**である。

## 8. リソース時分割（Folding）とモデル設計の関係

### 8.1 抽象原則: PE共有を前提としたレイヤー構成という発想

限られたFPGAリソース（DSP/LUT/BRAM）でモデル全体をハードウェア化するには、「全レイヤー分の演算器を空間的に敷き詰める」のではなく、**同じ物理的な演算器（Processing Element, PE）を複数のレイヤー・複数のタイルにわたって時分割（time-division/folding）で再利用する**ことが不可欠になる。この時分割には、大きく分けて2つの粒度がある。

1. **レイヤー内部の折り畳み（FINN流、`mvau.hpp`の`SIMD`/`PE`）**: 1つのレイヤーの行列演算サイズを、実装した並列度（`SIMD×PE`）のタイルに分解し、`NF×SF`回だけ同一のMACアレイを時分割で再利用する。
2. **レイヤー横断的な共有（SkyNet流、`#pragma HLS ALLOCATION instances=<func> limit=1 function`）**: 演算パターン（例: 全てのConv層が同じカーネルサイズ・同じ構造）が同一なら、レイヤーごとに別々のハードウェアを合成するのではなく、**1つのPE関数だけを合成し、全レイヤーがそれをチャンネル数・タイル数を変えながら順番に呼び出す**ことで、レイヤー数が増えても面積が増えないようにする。

この2つの軸は直交しており、両方を組み合わせられる（`claudes/design_philosophy.md` §2）。**モデル設計段階でこの時分割を見越しておくと、HLS実装が大きく単純化される**——具体的には「同じ演算パターン（同じカーネルサイズ・同じ演算子種別）のレイヤーをできるだけ揃える」ことが、レイヤー横断共有を成立させるための前提条件になる。

### 8.2 具体例: SkyNetのレイヤー横断PE共有

SkyNetは`net_hls.cc:805-808`で`CONV_1x1`・`DW_CONV_3x3`という2つの関数に対して`ALLOCATION instances=... limit=1 function`を指定し、dw1〜pw7までの全レイヤー（十数回の呼び出し）がこの2つの物理PE（32出力ch並列×16入力ch積和木＝512MAC、および32chの3x3積和木）を時分割で共有する（`claudes/skynet_explained.md` §5）。これが成立している理由は、**モデル側がすべてのブロックを「depthwise 3x3 conv + pointwise 1x1 conv」という統一されたBundle構造（`models.py:49-58`の`conv_dw`関数）で構成しているため**であり、もしモデルが層ごとに全く異なるカーネルサイズ・演算子種別を持っていたら、この共有は成立しない。**「レイヤーの演算パターンの種類数をできるだけ少なく（理想的には1〜2種類に）保つ」ことが、モデル設計段階でリソース時分割の効果を最大化するための直接的な指針になる。**

### 8.3 PE共有を前提としたレイヤー構成の考え方（モデル設計側の指針）

- **演算子の種類を絞る**: Conv（通常/depthwise/pointwise）、活性化、正規化、（必要なら）Upsampleの4〜5種類程度に絞り、種類が増えるたびに専用PE関数が1つ増える（＝面積が増える）ことを意識する。
- **カーネルサイズ・ストライドのバリエーションを絞る**（§4.2と同じ指針がここでも効いてくる）。
- **チャンネル数の刻み幅を統一する**（§4.1のPE数/SIMD数の倍数という制約と合わせて、全レイヤーが同じ並列度のPEを使い回せるようにする）。
- **Encoder/Decoderの対称性を意図的に維持する**（§7.2）ことで、AutoEncoder全体としてPE共有の対象範囲をさらに広げる。
- 一方で、**レイヤー横断共有（面積優先）とレイヤーごとの専用パイプライン（FINN流、スループット優先）はトレードオフである**ことも忘れてはならない——全レイヤーが1つのPEを共有すると、レイヤーは逐次実行になりスループットは下がる。AutoEncoderの推論頻度・レイテンシ要件とターゲットデバイスのDSP/LUT予算を見て、モデル設計段階から「どこまで演算パターンを統一するか」を意識的に選ぶべきである（`claudes/design_philosophy.md` §2 が示す通り、「どちらが正解」ではなく両方の引き出しを持っておくことが重要）。

## 9. Hardware-Aware Co-design

### 9.1 抽象原則: モデル探索とハードウェア実装は双方向であるべき

**Co-design（協調設計）**とは、モデルアーキテクチャの探索とハードウェア実装の設計を、どちらか一方を固定した上でもう一方を最適化する（例えば「精度最優先でモデルを決めてから、後でハードウェアに押し込む」）のではなく、**両者を同時に、あるいは反復的に調整しながら最適化する**設計手法を指す。SkyNetの元論文（DAC'19: arXiv:1904.04421、ICML'19workshop: arXiv:1905.08369）が提案する**"Bi-Directional Co-Design"**は、この考え方を明示的な方法論として体系化したものであり、以下の2方向の情報伝達を持つ。

- **Bottom-up（ハードウェア→モデル）**: ターゲットデバイス上で実際に実装・計測したハードウェア効率の良い「基本構成単位（Bundle）」——SkyNetの場合は3x3 depthwise conv + 1x1 pointwise conv + BatchNorm + ReLU6——を先に定義し、レイテンシ・リソース使用量のモデル（コスト関数）を作る。この「ハードウェアで測定した既知のブロック」を、モデル探索が使える部品の集合として提供する。
- **Top-down（モデル→ハードウェア）**: 探索されたネットワーク全体（Bundleの積み木としての組み合わせ）を、精度とハードウェア指標（レイテンシ等）を単一の適合度関数`Fit = Accuracy + α × (Estimated_HW_metric - Target)`（α<0）で評価し、ハードウェア制約を満たさない候補を早期に切り捨てながら、group-based particle swarm optimization（PSO）等でネットワーク構造そのものを探索する。

この双方向性——**「ハードウェアの現実（何が効率よく実装できるか）をモデル探索の制約条件として組み込む」と同時に、「モデル側の要求（精度を保つために必要な構造）をハードウェア実装側が受け止める」という、両者が互いにフィードバックし合う設計プロセス**——こそが、Co-designという言葉の核心である（`claudes/skynet_explained.md` §10）。

### 9.2 一般化: 自分でCo-designを行うための実務的な進め方

SkyNet論文のような自動探索（PSO等によるNAS）のフレームワークを毎回構築するのは大掛かりだが、**Co-designの考え方そのものは、手動でモデルを設計する際にも適用できる**。以下は一般化した実務手順である。

1. **ハードウェア側の基本ブロック（Bundle相当）を先に決める**: ターゲットデバイスのDSP/BRAM予算、目標クロック、目標レイテンシから、「1つのPEでどれだけの演算パターンを実装するのが現実的か」を先に見積もる（§8）。
2. **その基本ブロックのハードウェアコストモデルを作る**: 「Conv 3x3、チャンネル数Cのdepthwise+pointwiseブロックを1段追加すると、追加で必要なBRAM/DSP/レイテンシがどれだけ増えるか」を、実際に小さな実装で計測するか、あるいはFINN-Rのバランシングアルゴリズム（Section 4.4、`claudes/finn_explained.md` §9）のような解析的な見積もり式を用意する。
3. **モデル探索（あるいは手動設計の候補比較）に、このハードウェアコストを明示的な制約・ペナルティとして組み込む**: 精度だけでなく、「推定レイテンシ・推定リソース使用量が予算内に収まるか」を候補選定の基準に加える。SkyNetがModel A（bypassなし）/B（単純bypass）/C（bypass+concat採用）を比較検討し、精度とパラメータ数（1.27MB/1.57MB/1.82MB）を両睨みで最終構成を決めた例（`claudes/skynet_explained.md` §10）は、この比較検討プロセスの具体例である。
4. **実装後、実測値でコストモデルを補正するフィードバックループを回す**: 実際にHLS合成・実機計測した結果が見積もりと乖離していれば、コストモデル（手順2）を更新し、次のモデル候補選定（手順3）に反映する。

### 9.3 Co-designの適用例: 量子化ビット幅探索もCo-designの一種

§2.2で述べた「学習側でビット幅を総当たり探索し、その結果をHLSの型定義として埋め込む」（SkyNetのMLSys'20論文Table 7）というプロセスも、実は**Co-designの一種**である——ビット幅という「ハードウェアコストに直結するモデルパラメータ」を、精度指標と突き合わせながら選定しているという点で、Bi-Directional Co-Designの縮小版（探索空間をビット幅のみに絞った特殊ケース）と見なせる。この意味で、Co-designは「ネットワーク構造そのものを探索する大掛かりな手法」に限らず、**「量子化ビット幅」「活性化関数の選択」「スキップ接続の配置」といった個々の設計判断のレベルでも適用できる考え方**であり、本ドキュメントの§2〜§8で述べてきた各指針（ビット幅選択、演算子選択、次元設計、スキップ接続の緩和策）はすべて、この小さなCo-designの積み重ねとして位置づけられる。

### 9.4 参考: Hardware-Aware NAS（学術的な発展形）

より体系的な自動化を目指す学術分野としてHardware-Aware Neural Architecture Search（HW-NAS）があり、PE配列サイズ・dataflow・バッファサイズといったアクセラレータ側のパラメータとネットワーク構造を同時に探索する研究（例: EDD、arXiv:2005.02563、"Efficient Differentiable DNN Architecture and Implementation Co-search"）や、探索と評価を高速化するベンチマーク（HW-NAS-Bench, arXiv:2103.10584）が存在する。これらは「モデルとハードウェアパラメータの共探索がPareto効率的な解を生む」ことを示す学術的裏付けとなっており（同じ精度制約下でエネルギー消費を最大2倍改善できたという報告もある）、SkyNetのBi-Directional Co-Designはこの学術潮流の一つの実例と位置づけられる。**ただし、これらのHW-NASフレームワーク自体は本リポジトリでは未調査であり、具体的な適用方法（特にオンチップ完結・手書きHLSという制約下での適用可否）は確認できていない**点を明記しておく。

## 10. チェックリスト

モデルを設計・選定する際、実装前に確認すべき項目を以下にまとめる。§1〜§9で述べた抽象原則の実務的な落とし込みとして使うことを想定している。

### 10.1 全体設計・メモリ予算

- [ ] モデル全体の量子化後の重みサイズ（全層合計）を見積もったか（§1.6, §4.3）
- [ ] スキップ接続・分岐で保持が必要な中間活性化バッファの合計サイズを見積もったか（§1.6, §6）
- [ ] 上記の合計が、ターゲットデバイスのBRAM/URAM総量に対して十分な余裕を持って収まっているか（オンチップ完結を狙う場合は必須）
- [ ] 入力解像度・初期層のチャンネル数が、ラインバッファ方式で扱える規模に収まっているか（§4.3）

### 10.2 演算子選択

- [ ] 活性化関数はReLU系（ReLU/ReLU6等）を優先し、値域が有限に制約されているか（§3.1）
- [ ] Softmax/LayerNorm/GELU等の非線形処理を含む場合、それをFPGA上で実装する必要が本当にあるか（ホスト側へ逃がせないか）検討したか（§3.2）
- [ ] 正規化層（BatchNorm等）は畳み込み直後に固定配置し、推論時に前段へfuseできる構成になっているか（§5.1）
- [ ] Depthwise Separable Convを使う場合、pointwise（1x1）側のチャンネル数が小さすぎて演算強度が落ちすぎていないか（§3.3, §1.2）

### 10.3 量子化

- [ ] per-channel/per-tensorのどちらで量子化するか、層ごとに意図的に決めたか（§2.1）
- [ ] 各層のビット幅（`ap_fixed`の総ビット幅・整数部ビット幅）を、実測の値域分布に基づいて決めたか（§2.2）
- [ ] QAT（量子化認識学習）を、目標ビット幅に対して実施する必要があるか判断したか（§2.3）
- [ ] 丸め・requantizeのタイミングを層タイプ（積和の累積深さ）ごとに意識的に統一したか（§2.4）
- [ ] 低〜中ビット幅の中間活性化に対して、閾値ベースrequantize（乗算器レス）を適用できないか検討したか（§2.5）
- [ ] 高ビット幅（12bit以上）の層（特に出力層・再構成誤差を扱う層）は、閾値方式ではなく算術的requantizeに回したか（§2.5）

### 10.4 次元・形状設計

- [ ] チャンネル数が、想定するPE数/SIMD数の倍数になっているか（端数パディングが最小限か）（§4.1）
- [ ] カーネルサイズ・ストライドのバリエーションを絞ったか（§4.2）
- [ ] パディングは対称（same相当）に統一したか（§4.2）
- [ ] 実行時に形状が変わる要素（可変長・データ依存分岐）を排除し、静的形状だけで構成したか（§1.4）

### 10.5 演算子融合・PE共有

- [ ] Conv+BN+Activationを1つの融合ユニットとして扱えるモデル構造になっているか（§5）
- [ ] レイヤー間で演算パターン（カーネルサイズ・演算子種別）をできるだけ統一し、PE共有（レイヤー横断的な時分割）を成立させやすくしたか（§8）
- [ ] Encoder/Decoderが対称構造を持つ場合、チャンネル数の遷移刻み幅を揃えて、PE構成をEncoder/Decoder間で再利用できる余地を残したか（§7.2, §8）

### 10.6 スキップ接続・分岐

- [ ] スキップ接続の分岐点の数を必要最小限に抑えたか（§6.3）
- [ ] バイパス経路（分岐から合流までの距離）をできるだけ短く設計したか（§6.3）
- [ ] バイパス経路上に追加の変換（チャンネル数調整等）を極力挟まない構成にしたか（§6.3）
- [ ] 合流演算を単純な加算・チャンネル連結のいずれかに限定したか（§6.3）
- [ ] メインパス側のレイヤーごとのレイテンシを見積もり、バイパスFIFOの深さ設計に着手する準備ができているか（§6.4、モデル側の緩和だけでは解決しない残課題であることを認識）

### 10.7 AutoEncoder/Encoder-Decoder特有

- [ ] Decoder側のアップサンプリングにTransposed Convolutionではなく、最近傍補間+convのような代替を検討したか（§7.1）
- [ ] Encoder/Decoderの対称性を、精度だけでなくハードウェア再利用性の観点からも意図的に維持したか（§7.2）

### 10.8 Co-design（設計プロセス全体）

- [ ] ハードウェア側の基本ブロック（PE構成、想定並列度）を、モデル探索・設計の前に大まかに決めたか（§9.2 手順1）
- [ ] 精度指標だけでなく、推定ハードウェアコスト（レイテンシ・リソース使用量）を候補比較の基準に含めたか（§9.2 手順3）
- [ ] 実装後の実測値を、次のモデル改良のフィードバックとして活用する仕組み（あるいは少なくとも心づもり）があるか（§9.2 手順4）
